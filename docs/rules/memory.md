# memory 写入规则

> **本文件作用**：定义 `mine/memory/` 的写入规则，覆盖 people（人物档案）和 events（事件档案）两类。
> **核心定位**：memory 存「发生过什么、谁是谁」——事实优先、细节优先、不归纳、不替用户下结论。
> **与 raw 的边界**：raw 存观点/思考；memory 存事实/经历/人际。一段话里既有事件也有思考 → 拆两份，互链。
> **关联**：[redlines.md](../redlines.md) · [raw.md](raw.md) · [../../mine/memory/people/template.md](../../mine/memory/people/template.md) · [../../mine/memory/events/template.md](../../mine/memory/events/template.md)

---

## 0. 一句话原则

> **事实层做记录员，主观层让用户自己写。**

---

## 1. memory 适用判断

进入 memory 的内容必须满足下列任一特征：

| 特征 | 落到 |
|---|---|
| 介绍一个人（关系 / 背景 / 相处模式） | `people/<姓名或代号>.md` |
| 含「人 + 时间 + 行为」的具体事件 | `events/<分类>/YYYY-MM-DD-<关键词>.md` |
| 同时含事件和思考 | 拆两份：events + raw，互相链接 |

**不进 memory 的内容**：
- 纯观点 / 偏好 / 模式抽象 → 走 [raw.md](raw.md)
- to-do / 提醒 → 不进 Selfdex
- 纯情绪发泄 → 不进 Selfdex

---

## 2. 事件分类（events 子目录）

**建议分类**（按需新建即可，不要为单条事件提前建空目录）：

- `events/工作/` — 项目、晋升、跳槽、职场冲突、重要会议
- `events/感情/` — 伴侣 / 恋爱关系节点
- `events/友情/` — 重要朋友、聚会、断交
- `events/亲情/` — 家人、家庭事件
- `events/爱好/` — 兴趣相关的关键经历
- `events/健康/` — 体检、就医、身体状态变化（偏身体）
- `events/心灵/` — 内心成长、心理状态变化、自我发现（偏精神/成长）
- `events/财务/` — 重大支出、投资、收入变化
- `events/学习/` — 课程、考试、证书、自学项目
- `events/生活/` — 日常生活点滴、随意记录、偶发性小思考（**详见 §13 专项规则**）

> 你可以根据自己的实际情况增减分类——只要保证「分类边界清晰、不会反复纠结归到哪里」。

### 分类判断要点

- **心灵 vs 健康**：身体相关（体检、就医、生理状态）→ 健康；精神/心理/成长相关 → 心灵
- **感情 vs 心灵**：和某个具体人的关系节点 → 感情；自我内心的成长 → 心灵
- **工作 vs 学习**：项目/职场/收入相关 → 工作；课程/证书/自学 → 学习
- **拿不准 → 反问用户，不要猜**

---

## 3. 文件命名

| 类型 | 格式 | 示例 |
|---|---|---|
| people | `<姓名或代号>.md` | `张三.md` |
| events | `YYYY-MM-DD-<事件关键词>.md` | `2026-04-26-离职面谈.md` |
| events（时间段不是单日） | `YYYY-MM-<事件关键词>.md` | `2025-08-某项目复盘.md` |

**关键词要求**：
- 简短可识别
- 用事件主体或最核心动作（不用「今天和谁聊了什么」这种日记式命名）
- 同一天多个事件 → `YYYY-MM-DD-<关键词1>.md`、`YYYY-MM-DD-<关键词2>.md`

---

## 4. people 写入规则

### 4.1 字段填写责任划分

| 字段 | 谁来写 |
|---|---|
| 基本信息（姓名、关系、认识时间、当前状态） | AI 可起草，用户确认 |
| 背景 | AI 可起草，用户确认 |
| 相处模式 | **🔒 用户主笔，AI 不替写** |
| 重要节点（事件链接表） | AI 可填（基于已有 events 文件） |
| 用户的私人备注 | **🔒 AI 永远不写** |

### 4.2 重复人物处理

写入前必须先检查 `mine/memory/people/` 是否已存在该人的档案：
- **已存在** → 追加更新（不要新建文件，不要覆盖原有内容）
- **不存在** → 新建

### 4.3 关系字段措辞

- 用客观描述（「前同事 / 朋友 / 伴侣」），不用评价性措辞（「很好的朋友 / 不靠谱的同事」）
- 评价由用户在「私人备注」自己写

### 4.4 INDEX 同步

写入 / 更新 people 后必须更新 `mine/memory/INDEX.md` 中的人物表格（姓名 / 关系 / 文件 / 备注）。

---

## 5. events 写入规则

### 5.1 字段填写责任划分

| 字段 | 谁来写 |
|---|---|
| 元信息（日期、分类、涉及人物、地点） | AI 可起草 |
| 事件经过 | AI 可起草，用户确认/补充 |
| 关键细节 | AI 可起草（保留原话、表情、数字等具体细节） |
| 当时的我 | **🔒 AI 只能复述用户原话，不替用户加感受** |
| 后续思考 | AI 可补充 raw 链接 |

### 5.2 事件经过的写法

- **还原现场**：谁说了什么、做了什么、当时的情境
- **按时间顺序**，不要总结、不要因果归纳
- **能写多详细写多详细**，不压缩
- 用户原话直接引用（保留语气、用词），不改写

### 5.3 关键细节

容易被时间冲淡、但用户想留下来的：
- 一句原话（用引号）
- 一个表情、一个动作
- 一个具体数字、时间点、地点
- 一个标志性的对话片段

### 5.4 「当时的我」字段（红线区）

- 🔒 AI **只能复述用户原话**
- 用户没说过的感受，AI 不能补
- 用户说「当时我有点烦」→ AI 写「当时我有点烦」，不写「当时我感到强烈的挫败感」
- 用户没明确说当时的感受 → 留空 + 标注「待用户补充」

### 5.5 涉及人物的链接

事件元信息中的「涉及人物」必须用相对路径链接到 `people/<人名>.md`：
```markdown
- **涉及人物**：[张三](../../people/张三.md)、[李四](../../people/李四.md)
```

如果对应 people 文件不存在 → 提醒用户：「涉及人物 X 还没有人物档案，要不要先建？」

### 5.6 双向链接

事件 ↔ 相关人物 ↔ 相关思考（raw / compiled），三方互链：
- events 文件 → people 文件（涉及人物字段）
- events 文件 → raw 文件（后续思考字段）
- people 文件 → events 文件（重要节点表格）
- raw 文件 → events 文件（如果思考起源于具体事件）

---

## 6. 「事件 + 思考」拆分规则

如果用户一段话里既描述了事件、又表达了观点 / 反思 / 模式抽象：

```
事件部分（事实 / 时间 / 人物 / 经过）→ events/<分类>/YYYY-MM-DD-<关键词>.md
观点部分（"我意识到 / 我以后要 / 这让我想到"）→ mine/raw/<月度>.md
两边互相链接
```

**判断口诀**：
- 能用「X 月 X 日，谁谁谁做了什么」开头复述 → events
- 能用「我认为 / 我意识到 / 我倾向」表达 → raw

> ⚠️ 不要把事件和观点混在一篇 events 文件里。事件文件是事实记录，观点会污染它的「原貌还原」价值。

---

## 7. INDEX 同步（硬性要求）

每次写入 memory 后必须检查并更新 `mine/memory/INDEX.md`：

- 新增人物 → 在「人物档案」表格加一行
- 新增事件分类（之前没有的子目录）→ 在「分类」列表加一行
- 单条事件**不**逐条加索引（数量会爆炸）

> ⚠️ 写完 memory 不更新 INDEX，会导致后续 AI 检索按 INDEX 走时找不到对应人物。**这是硬性步骤。**

---

## 8. 写入流程（standard procedure）

```
1. 判断归属
   - people 还是 events？
   - events 是哪个分类？（用户原话 + §2 判断要点 + 拿不准就反问）
   - 是否已存在同名 people / 同主题 events？

2. 套模板
   - people → mine/memory/people/template.md
   - events → mine/memory/events/template.md
   - 复制后再填，不直接改 template.md

3. 起草（遵守责任划分 §4.1 / §5.1）
   - 红线字段（私人备注 / 相处模式 / 当时的我）必须留空 + 标注「待用户补充」
   - 涉及人物自动加链接

4. 通用确认协议
   - 列【内容 / 路径 / 分类 / 涉及人物 / 哪些字段留空】给用户确认
   - 提示用户哪些字段需要他亲自补
   - 用户回「确认」或「OK」才落盘

5. 落盘
   - edit_file 写入对应路径
   - 同步更新 INDEX（§7）
   - 双向链接补全（§5.6）

6. commit（不 push）
   git -C <SELFDEX-PATH> add -A && git -C <SELFDEX-PATH> commit -m "memory-people: <姓名>"
   或
   git -C <SELFDEX-PATH> add -A && git -C <SELFDEX-PATH> commit -m "memory-event: <主题>"

7. 告知用户
   "✅ 已记录到 <路径>，本地已 commit。
    [需要你手动补的字段]：<列出留空字段>。
    记得定期 push。"
```

## 9. 自查清单（交付前过一遍）

### 9.1 归属
- [ ] 这是 people 还是 events？还是两者都要拆？
- [ ] events 分类选对了吗？心灵/健康、感情/心灵、工作/学习有没有混？
- [ ] 用户的「思考性」内容是否误塞进了事件经过？

### 9.2 红线字段
- [ ] 「私人备注」是否留空？
- [ ] 「相处模式」是否仅复述了用户原话，没有 AI 加工？
- [ ] 「当时的我」是否仅复述用户原话，没有添加未说过的感受？

### 9.3 事实保真
- [ ] 用户原话是否用引号保留？
- [ ] 时间顺序是否被打乱成因果总结？
- [ ] 关键细节（数字 / 表情 / 标志性对话）是否保留？

### 9.4 链接
- [ ] 涉及人物都链接到了 people 文件？
- [ ] 涉及人物没有人物档案的，是否提醒用户建？
- [ ] events 与 raw 的双向链接是否补全？

### 9.5 落盘前
- [ ] INDEX 是否更新？
- [ ] 重复人物是否走的「追加更新」而非新建？
- [ ] 文件名命名规范？（YYYY-MM-DD-关键词）

## 10. 反模式

- ❌ **替用户感受**：用户没说当时的心情，AI 自己写「当时她应该很难过」
- ❌ **事件 + 观点混写**：把「我意识到 X」塞进事件经过里
- ❌ **总结代替还原**：把「今天和 A 聊了 3 小时」代替具体的对话内容
- ❌ **评价性措辞**：在关系字段写「很好的朋友 / 不靠谱的同事」
- ❌ **重建已有人物档案**：已存在的 people 文件，又新建一份覆盖
- ❌ **写完不更新 INDEX**：导致检索按 INDEX 走时漏掉
- ❌ **强行分类**：不在已有分类里的事件硬塞，而不是反问用户

---

## 11. 错误处理

用户指出归属错误 / 字段越界 / 感受补错时：

1. 立即承认并改正
2. 越界字段（私人备注 / 当时的我等）立刻清空 + 标「待用户补充」
3. 分类错误 → 用 `git mv` 移动文件，更新所有引用本文件的链接
4. 同批写入是否存在同类错误，一并检查

## 13. 生活分类专项规则（`events/生活/`）

### 13.1 定位

`events/生活/` 用于记录日常生活中的点滴与偶发性小思考：随意的、非系统的、不深度的。与其他分类的核心区别：

- **不要求完整还原现场**，只提炼总结性描述，体现当月状态即可
- **不保留全部内容**，只留有代表性的片段
- **按月归档**，文件命名格式：`YYYY-MM-生活.md`（同月内容追加到同一文件，不新建）

### 13.2 文件结构

```
events/生活/
  2026-05-生活.md
  2026-06-生活.md
  ...
```

### 13.3 写入要求

- **总结性描述为主**：不逐条还原，用几句话概括这段时间的生活状态、情绪基调、典型场景
- **保留有质感的细节**：一句原话、一个具体场景、一个有趣的小发现——能体现「这个月的我」的细节留下，流水账删掉
- **不强求结构**：生活记录天然随意，不需要套 events 标准模板，自由段落即可
- **当时的我**：🔒 同样遵守红线——AI 只能复述用户原话，不替用户加感受

### 13.4 交叉更新

生活记录中如果涉及其他分类的人或事件，**同步更新**对应文件：
- 涉及已有 people 档案的人 → 在对应 `people/<姓名>.md` 的重要节点表格补一行
- 涉及其他 events 分类（如感情、工作）的节点 → 在对应 events 文件补链接或追加内容
- 双向链接规则同 §5.6

### 13.5 raw 提炼提示（AI 主动识别）

生活记录中如果出现**可以凝练为观点/思考**的内容（满足 raw.md §1 收录标准），AI 必须：

1. **不直接写入 raw**
2. 在交付生活记录后，单独列出：
   ```
   💡 以下内容可能值得归纳到 raw，是否需要？
   - [简短描述观点内容]
   ```
3. 等用户确认后，再走 raw 写入流程

**判断口诀**：能用「我认为 / 我意识到 / 我倾向」表达的 → 提示归纳到 raw；只是「发生了什么 / 当时怎样」的 → 留在生活记录里。

### 13.6 与其他分类的边界

| 内容 | 归到 |
|---|---|
| 日常生活点滴、随意小思考 | `events/生活/` |
| 深度系统的内心成长、自我发现 | `events/心灵/` |
| 身体健康相关 | `events/健康/` |
| 可凝练为观点的思考 | 提示用户 → `mine/raw/` |
