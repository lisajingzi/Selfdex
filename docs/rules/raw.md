# raw 写入规则

> **本文件作用**：定义 `mine/raw/` 与 `others/raw/` 的写入规则，包括「收/不收」判定、来源可追溯、观点提炼、独立可读、完整性自查。
> **何时被调用**：AI 接到「记到 Selfdex」「提炼这段对话到 raw」「整理这篇文章」等任务时，先看 [flow.md](../flow.md)，再读本文。
> **关联**：[redlines.md](../redlines.md)（红线）· [templates/raw-mine.md](../templates/raw-mine.md) · [templates/raw-others.md](../templates/raw-others.md) · [memory.md](memory.md)（事件 vs 观点的边界）

---

## 0. 一句话原则

> **每一条写入 raw 的内容，要么撞到了你，要么能被你将来用上；每一条结论，都要能回到原文。**

---

## 1. 收 / 不收 判定

### ✅ 收（满足任一即可）

| 信号 | 含义 | 例子 |
|---|---|---|
| 🔥 **有刺激** | 引发明显反应：共鸣 / 不爽 / 惊讶 / 想反驳 | 一句话让你想拍桌子 |
| 🔁 **反复出现** | 同一观点在不同来源反复撞见 | 三本书都讲同一个谬误 |
| 🔧 **可复用** | 将来在某个具体场景能拿出来用 | 一个判断框架、一个具体方法 |

### ❌ 不收

- 一次性资讯（新闻 / 热点 / 八卦）
- 可随手搜到的客观事实（除非你有自己的反应）
- 纯情绪发泄（去日记 app）
- To-do / 提醒（去任务管理器）

### 🚦 3 秒决策

```
能 3 秒说出"它撞到了我什么"？
  ├─ 能 → 收
  └─ 不能 → 它能解决我未来某个具体场景吗？
              ├─ 能 → 收
              └─ 不能 → 跳过
```

> **宁可漏收一条值得收的，也不要乱收一条不值得收的。**

## 2. mine/raw vs others/raw

| 起点 | 落到 | 模板 |
|---|---|---|
| 我自己（灵感 / 复盘 / 提问） | `mine/raw/` | [raw-mine.md](../templates/raw-mine.md) |
| 外部（文章 / 书 / 播客 / 他人话） | `others/raw/` | [raw-others.md](../templates/raw-others.md) |

**判断口诀**：起点是「我」→ mine；起点是「外部内容」→ others。

边界场景：
- 我的想法但起点是别人一句话 → mine（在内容里提一嘴「由 XX 触发」）
- 别人的内容但我反应强烈 → others（反应写充分）
- 自己复盘里夹带某文章观点 → mine（外部观点加引号并标来源）

> ⚠️ `others/raw/` 通常必须由用户补「我的反应」。AI 不能代笔。
> **例外**：划线类素材（如电子书划线、网页高亮）——划线本身即代表认可，无需额外填写「我的反应」。

## 3. 文件命名与归档位置

> ⚠️ **mine 和 others 命名规则不同，不要混用。** 写入前先确认是哪条轨道。

### 3.1 mine/raw（按时间组织）

**命名**：`YYYY-MM-DD-<主题关键词>.md`（日期在前，因为 mine 是时间线）

**归档位置**：
- `mine/raw/YYYY-MM.md`：同月零散观点合并到月度文件
- `mine/raw/conversations/YYYY-MM-DD/<主题>.md`：信息量大的独立对话单独成文
  - 子目录用对话发生日期，同一天多个对话文件放同一子目录下
  - 好处：编译时可按月/日筛选，避免根目录堆积

### 3.2 others/raw（按来源组织）

**命名**：`<来源标题>-YYYY-MM-DD.md`（标题在前，日期在后，因为 others 是按「哪本书 / 哪篇文章」组织的）

- 来源标题用原标题，去掉书名号 `《》`、引号、副标题
- 日期为整理日期（不是原文发表日期）
- 示例：
  - ✅ `纳瓦尔宝典-2026-04-11.md`
  - ❌ `2026-04-11-纳瓦尔宝典.md`（这是 mine 的格式）
  - ❌ `《纳瓦尔宝典》-2026-04-11.md`（带书名号）

**归档位置**：直接放 `others/raw/` 根目录，不分子目录。

### 3.3 通用规则

- 不确定走哪条轨道时反问，**不要新开目录**
- 写入前先看目标文件是否已存在；存在就追加段落到原文件，**不要新开文件**
- 主题关键词 / 来源标题简短可识别

## 4. 来源可追溯（提炼对话 / 长原文时强制）

### 4.1 来源隔离

**允许使用**：
- 用户本次指定的原文 / 对话 / 文档
- 用户明确要求同时参考的其他 Selfdex 文件
- 用户在当前对话中补充的说明

**禁止使用**：
- AI 对用户的历史印象
- 其他未被指定的历史对话
- 其他 LLM 的观点
- AI 觉得「合理」的补全
- 为了让总结更完整而加入的推测性解释

引入其他来源时必须显式标注：「来自本次原文」「来自 Selfdex 其他文件」「AI 推断」「待用户确认」。

### 4.2 逐条可追溯

所有结论 / 标签 / 关键词，都必须能指回原文：
- 哪段日期 / 哪一节 / 哪轮对话
- 附原话摘录（确认阶段用，写入阶段去掉）
- 多段综合得出的结论 → 列出多个来源点

### 4.3 低置信度降级

不确定是否真为用户观点时，按优先级降级：
1. 删除该条
2. 标注「可能的理解，待确认」
3. 改写为「对话中出现过的观点」而非「用户认为」
4. 直接询问用户

**禁止为了文档完整而硬写。**

---

## 5. 归属四分类（提炼对话时强制标注）

每条提炼出来的观点必须明确标注归属，从下面四类选一：

| 标签 | 定义 | 判断依据 |
|---|---|---|
| **【用户原本观点】** | 对话开始前已持有 / 主动提出 | "我认为 / 我一直觉得 / 我原来以为" |
| **【对话中接受的新理解】** | 对话过程中形成的新理解或修正 | "我明白了 / 你的意思是 / 所以其实是" |
| **【用户认可的他人说法】** | 来自第三方（朋友 / 书 / 播客）且用户认可 | 用户引用并表态认同 |
| **【用户反对/保留的观点】** | 对话中出现但用户明确反对或保留 | "我不同意 / 这个不对 / 这里有问题" |

> 🔒 四分类标注是硬性要求，不允许混写或省略。
> 🔒 只是「LLM 说过」≠「用户认同」。AI 单方面说的话不能写成用户的新理解。

---

## 6. 观点措辞规范（独立可读原则）

### 6.1 独立可读

每条观点必须做到「脱离原文上下文也能被陌生读者理解」。

❌ **不允许**：
- 「他所谓的独特」「这件事」「那个人」「上次那个问题」等指代性表述
- 「关于工具化」这类只有标签没有命题的写法
- 依赖前一条观点才能读懂的连贯叙述

✅ **要求**：
- 每条观点 = 「概括标签」+「完整命题（含主语、前提、结论）」
- 每条独立成段，删任意一条不影响其他

### 6.2 保留判断的锋利度

- 用户反对什么 → 写清楚反对什么
- 用户认可什么 → 写清楚认可的边界
- 用户的判断、定性、价值排序 → 完整保留

❌ 不允许把「我不认同 X」软化为「X 是一种观点，也有道理」。

### 6.3 关键词保留原话

观点措辞可以重写得更清晰，但：
- **不能扭曲原意**
- 用户用的核心词必须保留
- 重写后的版本要能映射回原话——用户读完每条都应能确认「是的，我是这么想的」

### 6.4 结论描述的详细度

每条结论必须满足「未来再看时能独立理解」：
- **不能过于简洁**：不能只写关键词或短语
- **包含足够语境**：读者不需要回到原文就能理解
- **保留具体表达**：精彩的比喻、命名、原创表述要保留
- **避免抽象化丢失信息**

## 7. 出处与正文分离

- 观点正文中**不**混入「出处第几段」「轮次 X」等元信息
- 出处用斜体小字附在每条末尾（一行），便于校验，但不喧宾夺主
- 出处只标足以定位的最小信息（事件名 / 段落主题 / 原话关键词）

**例外：用户原话直接输入时，不需要写出处行。**

格式：
```markdown
#### N. 关于「XX」的观点 【用户原本观点】
完整命题正文……

*出处：原文中我对 XX 事件的复盘段落。*
```

---

## 8. 完整性自查（提炼前强制执行）

完成初稿后必须做完整性校验：

1. **逐轮扫描**：对 LLM 对话，逐轮检查用户发言，确认每一轮中用户表达的观点、情绪、发现都已被覆盖
2. **对照清单**：从头到尾再过一遍原文，标记是否有遗漏
3. **数量感知**：30 轮对话却只有 5-6 条结论 → 大概率有遗漏，重新检查
4. **边缘内容不丢弃**：哪怕看起来「不重要」，只要是用户明确表达的，都应保留
5. **比喻和命名保留**：用户创造的比喻、命名、类比是思想的重要载体，**不能在总结中丢失**

校验输出：
- 发现遗漏 → 补充后重新交付
- 不确定是否值得保留 → 标注「边缘内容，是否保留？」交给用户

---

## 9. tags 与 INDEX 同步（硬性要求）

每次写入 raw 后必须检查并更新：

1. **本文件顶部 frontmatter `tags`**：根据本次内容补充关键词
2. **同月度 md 文件顶部的标签汇总**（如果存在）：把本次新增的标签合并进去
3. **`mine/raw/INDEX.md`**（如果存在）：补一行链接

> ⚠️ 写完 raw 不更新 tags，会导致后续检索完全找不到。**这是硬性步骤，不是可选项。**

## 10. 写入流程（standard procedure）

```
1. 判断归属
   - mine/raw 还是 others/raw？
   - 是新建文件，还是追加到现有月度文件？

2. 收/不收判定
   - 走 §1 三秒决策
   - 不收 → 直接告知用户原因，结束

3. 提炼（如果是长原文/对话）
   - 走 §4 来源可追溯
   - 走 §5 归属四分类
   - 走 §6 独立可读
   - 走 §8 完整性自查

4. 起草 + 通用确认协议
   - 套对应模板
   - 列【内容（含归属标签）/ 路径 / 分类】给用户确认
   - 提炼任务额外附「出处校验表」（轮次 + 原话摘录）
   - 用户回「确认」或「OK」才写入

5. 落盘
   - edit_file 写入对应路径
   - 写入版本去掉出处校验表，保留斜体出处行
   - 同步更新 tags / INDEX（§9）

6. commit（不 push）
   git -C <SELFDEX-PATH> add -A && git -C <SELFDEX-PATH> commit -m "raw: <主题>"
   commit 类型：raw / others-raw

7. 告知用户
   "✅ 已记录到 <路径>，本地已 commit，记得定期 push。"
```

## 11. 自查清单（交付前过一遍）

### 11.1 来源
- [ ] 这条总结是否只来自本次指定来源？
- [ ] 有没有混入历史印象？
- [ ] 有没有混入其他文件、其他对话、外部知识？
- [ ] 如果混入了，是否已经标注来源？

### 11.2 归属
- [ ] 这句话到底是用户的观点，还是 LLM 的观点？
- [ ] 是用户原本就这么想，还是对话后形成的新理解？
- [ ] 用户是认同、反对、保留，还是仅仅在讨论？
- [ ] 四分类标签是否每条都标了？

### 11.3 证据
- [ ] 每一条总结是否都有原文依据？
- [ ] 是否能展示对应原文？
- [ ] 用户追问「这条从哪里来」，能否立刻指出？

### 11.4 表达
- [ ] 是否使用了过度确定的表达？
- [ ] 是否把低置信度内容写得像事实？
- [ ] 是否为了让结构更完整而补写了用户没说过的话？
- [ ] 是否保留了用户原文中的矛盾、摇摆、疑问？

### 11.5 独立可读
- [ ] 每条观点单独拎出来，陌生读者也能看懂吗？
- [ ] 有没有「这个」「那个」「他所谓的」等指代词？
- [ ] 用户的核心原词（比喻 / 命名 / 锋利的词）保留了吗？

### 11.6 完整性
- [ ] 原文每一段用户明确表达过态度的内容都覆盖了吗？
- [ ] 有没有因为「看起来不重要」就丢掉？

### 11.7 落盘前
- [ ] tags 是否已更新？
- [ ] 月度文件顶部标签汇总是否同步？
- [ ] INDEX 是否已补链接？
- [ ] 是否新开了不必要的文件？应该追加到现有月度 md 吗？

## 12. 反模式

- ❌ **追求完整收录**：把整篇文章原文 copy 进来
- ❌ **现场过度分类**：raw 阶段就纠结「这个该归到哪个标签下」
- ❌ **延迟归档**：想「等我有时间整理一下再放进去」
- ❌ **替代日记**：把 Selfdex 当朋友圈写心情
- ❌ **跳过完整性自查**：原文 30 轮只总结 5 条就交付
- ❌ **指代词偷懒**：写「他所谓的 X」而不是写清楚 X 是什么
- ❌ **把 AI 建议写成用户观点**：LLM 单方面说的话写成「用户认为」
- ❌ **写完不更新 tags / INDEX**：导致检索失效

---

## 13. 错误处理

如果用户指出某条总结没有来源 / 来源错误 / 归属错误，必须：

1. 立即承认该条不可靠
2. 回到原文重新定位依据
3. 找不到依据 → 删除该条 或 改为「AI 推断，待确认」
4. 检查同批总结中是否存在同类错误
5. 必要时重做整段总结

**不允许**：
- 辩解「这是合理推断」
- 用用户过往 profile 证明当前总结
- 在没有原文依据的情况下继续保留
- 把错误轻描淡写为措辞问题

---

## 14. 一句话兜底

> 提炼不是「把内容写得像我」，而是「让每一句结论都能回到我自己说过的话」。
