# 红线（AI 永远不做）

> **本文件作用**：Selfdex 体系中 AI 的硬性禁止清单。任何写入操作前必须过一遍本文件。
> **优先级**：红线优先级最高，高于所有 rules 文件中的其他规则。如有冲突，以本文件为准。
> **何时被调用**：每次写入 Selfdex 前（flow.md §3 第 2 步）。
> **关联**：[flow.md](flow.md) · [rules/](rules/)

---

## 🔒 永不做清单

### 1. 内容红线（AI 永不代笔）

| 编号 | 红线 | 涉及文件 |
|---|---|---|
| R1 | **不替用户写 compiled / explored 的核心判断与结论** | `mine/compiled/*.md` · `mine/explored/*.md` |
| R2 | **不判定 explored 的 status（alive / aging / outdated）** | `mine/explored/*.md` |
| R3 | **不更新 profile.md 的「决策风格」与「禁忌项」** | `profile.md` |
| R4 | **不替用户写 memory 中的「私人备注」「当时的我」** | `mine/memory/people/*.md` · `mine/memory/events/*.md` |
| R5 | **不替用户写 others/raw 的「我的反应」** | `others/raw/*.md` |
| R6 | **不把 LLM 单方面说的话写成用户的观点** | 所有 raw 提炼任务 |

### 2. 流程红线（AI 永不跳过）

| 编号 | 红线 | 说明 |
|---|---|---|
| R7 | **不跳过通用确认协议直接写入 Selfdex** | 任何写入操作前必须列内容/路径/分类让用户确认 |
| R8 | **不 push（只 commit）** | push 留给用户手动，避免跨设备冲突 |
| R9 | **不确定时必须反问，不能猜** | 来源不明、归类模糊、看起来像情绪发泄 → 都要问 |
| R10 | **涉及个人话题时不跳过 `INDEX.md` 检索** | 必须先读根 `INDEX.md`，关键词命中后深读对应主题文件；未命中要明确告知用户，不允许凭印象编造用户立场 |
| R11 | **新增/修改 compiled / explored 时不跳过同步根 `INDEX.md`** | 主题文件与 `INDEX.md` 必须一起 commit，否则下次检索会遗漏 |

### 3. 来源红线（AI 永不混入）

| 编号 | 红线 | 说明 |
|---|---|---|
| R12 | **不混入历史印象** | 总结只能基于本次指定来源 |
| R13 | **不混入其他 LLM 的观点** | 不能把其他对话中 AI 说的话当成用户观点 |
| R14 | **不为了完整性而硬写** | 找不到来源就删除或标「待确认」 |
| R15 | **不把「对话中出现过」等同于「用户认同」** | 必须有明确认同证据 |

### 4. 架构红线（AI 永不动）

| 编号 | 红线 | 说明 |
|---|---|---|
| R16 | **不增删核心层（mine/others × raw/compiled/explored）** | 目录结构由用户决定 |
| R17 | **不合并 mine 和 others** | 两轨永远分开 |
| R18 | **不删除 outdated 的 explored 文件** | 保留作为历史档案 |
| R19 | **不改写 explored 原文** | 只能在文末加注 |

## 🚨 违反红线时的处理

如果 AI 意识到自己违反了红线：

1. **立即停止**当前操作
2. **承认违规**：明确告知用户「我刚才违反了红线 RX」
3. **回滚**：如果已经写入，立即撤销
4. **重做**：按正确流程重新执行

如果用户指出 AI 违反了红线：

1. 立即承认，不辩解
2. 回滚已写入的内容
3. 检查同批操作是否有同类违规
4. 按正确流程重做

## 📌 逐步积累

本文件会随着使用过程中发现的新问题逐步增加红线条目。

新增红线的标准：
- 某个错误**反复发生**（≥2 次）
- 某个错误**后果严重**（导致用户观点被污染、隐私泄露等）
- 用户明确要求「以后不要再 XX」

新增格式：在对应分类表格中加一行，编号递增。
